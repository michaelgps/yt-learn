## 🚢 总框架:把自己当"船长"，agent 当"船员"

他把成长路径分成几级，核心是**你越往上走，越不碰具体代码，越管方向**：

1. **搭船**：终端环境（见下）。
2. **招募+培训船员**：用记忆文件 + 技能，让 agent 懂你怎么干活。
3. **带一个船员干活**：规划 → 实现 → 验收，三段式。
4. **同时带多个船员**：并行多 session。
5. **请一个"大副"**：一个总管 agent 替你调度所有船员，你只谈大方向。

> 最值钱的一句心态：**别再像审 diff 的人，要像工程总监**——总监不审 PR，靠定流程和文化让团队保质量。对 AI 也该这样，否则"你每天能审的 diff"就成了你产出的天花板。

## 🧰 他的环境（终端为主，理由很实在）

| 工具 | 是什么 | 为什么用它 |
|------|--------|-----------|
| **WezTerm** | 终端模拟器 | 真跨平台（Win/Mac/Linux 一致）、可用 Lua 脚本配置 |
| **tmux** | 终端分屏/多会话管理 | 会话**持久**：手机/笔记本能连同一个会话接着干；并行跑多个 agent |
| **Neovim** | 编辑器 | 手不离键盘 = 不打断心流（他认为这是被低估的大事） |

> 他坦言：不喜欢终端也没关系，**视频讲的是思路，GUI 工作流照样适用**。

**四个 agent 工具他都常用**，且刻意保持"换谁都行"（因为格局变太快）：
- **Claude Code** —— 默认体验最顺、功能最全，但偶尔有 bug、可定制性一般。
- **Codex CLI** —— Rust 写的，更顺滑，开源（出问题能让它读自己源码自救）。
- **Pi** —— 极简、极可扩展，适合爱折腾的人。
- **opencode** —— TUI 顺滑、对接各种模型，开箱即用，模型无关首选。

## ✅ 能直接搬进你工作流的

- **全局记忆文件保持极简**（他只有 ~27 行）。它会塞进**每个** session 的系统提示，写多了 = 悄悄烧 token。用符号链接让 `CLAUDE.md` 和 `AGENTS.md` 指向同一文件，一份伺候所有 agent。
- **项目记忆 = 攒出来的，不是写出来的**：每次看到 agent 做错，纠正它、并让它把教训存进记忆文件。越用越懂这个项目。文件变臃肿时，**把"只在特定情况才用到"的内容挪进技能**（技能是"渐进披露"：平时只加载一行描述，真要用时才读全文，省 token）。
- **语音输入**（他几乎全语音；斯坦福论文：说话比打字快 3 倍）。本地用 **open-super-whisper**（免费开源、本地跑 whisper）。只有 URL/路径才打字。把常用专有名词写进转写的 initial prompt，识别率大涨。
- **规划别看"文字墙"，用可视化**：他的 **Lavish** 工具让 agent 生成一个**用项目自身设计风格的 HTML 原型**，你在浏览器里圈点批注、点选项做决定，再发回——比读一大段 markdown plan 高效太多。
- **验收别逐行审 diff**：他把 agent 产出的代码丢进自建流水线 **no mistakes**（开源）：自动建分支/commit → 在隔离 worktree 里 → 从 session 推断你的真实意图 → rebase 到 main 并提前解冲突 → **在全新上下文里做对抗式 review** → 端到端测试并**录证据（截图/录屏/日志）** → 更新文档 → lint → 提 PR → 一直盯到合并。它给的**风险评估**决定你要不要细看：低风险直接不看 diff。
- **让 agent 通宵干**：他的 **good night have fun**（开源）给个目标 + 停止条件（token/轮次上限）就循环到达标。例子绝妙：*"假装你是 7 岁小孩，端到端用这个 App，找出第一个让你困惑的可用性问题，修掉，再来一遍。"* 比 `/go` 命令好在能精确设上限，不会半夜烧光周配额。
- **多 agent 并行的 worktree 管理**：git worktree 能避免互相踩，但你得记住"我开了哪些、哪个在跑"——心智负担。他的 **treehouse** 自动管 worktree（关 tab 即释放、下次复用空闲的）。
- **终极：请个"大副"**：他的 **first mate**（开源）是个你对话的总管 agent，自动开 tmux tab、调 treehouse 建 worktree、跑 no mistakes、提 PR。你只谈方向。**瓶颈会从"AI 不够快"变成"你想不出该让它干嘛"**——这时该去找用户聊、研究竞争格局、定方向。

## 💡 最值得记住的反直觉结论（都有依据）

- **别太把"开发成本"当回事**（他写进了全局记忆）：模型是拿人类数据训练的，你问它"做个 3D 射击游戏要多久"，它答"几天几周几月"；但你让它**真做**，几分钟就给你能玩的版本。**模型不知道自己写代码比人快得多**，于是做技术决策时高估成本、倾向选"便宜但低质、难维护"的方案。要主动纠偏。
- **别信网上的高星技能**：他用 ProgramBench 测了那个 **177k 星的"Karpathy skills"** repo（其实**根本不是 Karpathy 写的**），结果是 **多花 5% token、效果还更差**。高星只说明流行，不等于有用。**没有严谨评测证明的"魔法技能"一律别装**——还有安全风险（技能能在你机器上跑任意命令、泄露密钥）。
- **工具的"对 agent 友好度"差别巨大**：用 GitHub **MCP server** 比用 CLI **多花 3 倍 token、2 倍延迟**。他为此定了一套面向 agent 的工具设计标准 **axi/ACI**（10 条原则，比如用紧凑输出格式比 JSON 省 ~40% token），并做了 GitHub axi、Chrome DevTools axi。**给 agent 配工具前先查它的效率。**
- **改 bug 从端到端复现开始**，别一上来写单测（AI 默认爱写单测，常常盖不住真实产品行为）。
- 小处见讲究：禁用 em-dash（`—`），因为一看就是 AI 味。
