## 🚢 一句话抓住全片：把自己当"船长"，AI 当"船员"

他用航海做比喻，把整套工作流拆成五级，**核心规律是：你越往上走，越不碰具体代码，越只管方向**。

1. **造船** —— 先把干活的环境搭好（终端那一套）。
2. **招募并培训船员** —— 让新来的 AI 搞懂"你这艘船怎么开、你喜欢怎么干活"。
3. **带一个船员干活** —— 规划 → 让它实现 → 你验收，三段式。
4. **同时带好几个船员** —— 并行开多个 AI 一起干。
5. **请一个"大副"** —— 一个总管 AI 替你调度所有船员，你只跟它谈大方向。

读这篇时记住这条线就行，下面每一节都是在往这条线上填东西。

## 第 1 级 · 造船：为什么他几乎所有事都在终端里做

你打开他屏幕第一眼就会发现：他基本不用图形界面，全在命令行里。他给了两个很实在的理由：

- **手不离键盘 = 不打断心流**。只要你的手一去摸鼠标，大脑就被迫切换一次注意力。终端工具天生为键盘设计，没有让你去碰鼠标的地方，所以能长时间保持专注。他说这件事被绝大多数人低估了。
- **同一套工作流哪里都能用，连手机都行**。这点下面 tmux 会展开。

他也很坦诚地说：**如果你就是不喜欢终端，没关系**——这条片子讲的是底层思路，图形界面照样能套用。

他这套环境是三件套：

| 工具 | 是什么 | 他为什么选它 |
|------|--------|------------|
| **WezTerm** | 终端窗口本身（终端模拟器） | 真正跨平台：Windows、Mac、Linux 上体验完全一致（他在微软被迫用 Win 时这点救了命）；还能用 Lua 脚本随意配置，改个配色当场热重载 |
| **tmux** | 在一个终端里开多个分屏/多个会话 | ① 一块屏拆成几格，一格跑 AI、一格开编辑器、一格自己敲命令；② **会话是持久的**：你断开、关掉，再连回来还是原样——甚至能从**笔记本或手机**连上同一个会话接着干。这就是"哪都能用"的底气 |
| **Neovim** | 键盘驱动的编辑器（Vim 的现代版） | 一切操作都用按键完成，光标移动、查找、删改都不碰鼠标，配合肌肉记忆"像鸟一样飞" |

> 他特意提醒：tmux 和 Neovim 装完默认都不好用，得花时间配置成自己顺手的样子。这是前期投入，别指望开箱即用。

## 第 2 级 · 招募并培训船员

### 先选船员：四个 AI 工具，他都常用

而且他**刻意保持"换谁都行"**，因为这行变化太快，下个月谁最强谁知道。所以他全片讲的东西都跟具体模型/工具无关。

- **Claude Code** —— 默认体验最顺、功能最全，是订阅 Anthropic 时基本唯一的实际选择。缺点：偶尔有 bug，可定制性一般。
- **Codex CLI** —— Rust 写的，用起来比 Claude Code 更顺滑；开源，出问题能让它**读自己的源码**找绕过办法。花哨功能少、不太能定制。
- **Pi** —— 哲学是"极简 + 极可扩展"，适合不想要任何臃肿、爱自己折腾的人。
- **opencode** —— TUI（文字界面）非常顺，几乎对接所有模型，开箱功能比 Pi 全。想要个"模型无关、拿来即用"的，选它。

片子后面他用 Claude Code 演示，因为大家最熟。

### 再培训：靠两样东西——记忆文件 + 技能

新来的 AI 是"白纸新兵"，不懂你的规矩，得 onboarding。他靠两样：

**① 记忆文件（memory file）**，分全局和项目两层。

- **全局记忆要极简**。他全局文件只有 **27 行**。为什么这么克制？因为这个文件的内容会被塞进**每一个** AI 会话的系统提示里——写多了，等于每次请求都在悄悄烧你的 token。所以这里只放**跨项目都成立的个人偏好**。他还有个 `opinions.md` 放观点，单独话题不展开。
  - 一个细节规则：**禁止用 em-dash（`—`）**。AI 被训练得默认爱用这个破折号，他一看就觉得"机器味"，写 PR 描述时不要。
  - 一条很值钱的规则：**做技术决策时，别太把"开发成本"当回事**。原因很有意思——你问前沿模型"做个能本地玩、带 AI 敌人的 3D 第一人称射击游戏要多久"，它会答"几天几周几个月"（因为它是拿人类数据训练的，这是**人类程序员**的估计）；但你让它**真去做**，几分钟就给你一个能玩的版本。**模型还没意识到自己写代码比人快得多**，于是做决策时高估了某些方案的成本，进而倾向于选"便宜但低质、不可扩展、难维护"的做法。这条规则就是去纠正这个偏见。
  - 还有一条：**修 bug 永远从"端到端复现"开始**——尽量按真实用户的使用方式复现，而不是 AI 默认爱写的单元测试（单测常常盖不住真正要守护的产品行为）。

- **项目记忆是"攒"出来的，不是一次写好的**。每次他看到 AI 在这个项目里做错了，就纠正它、并让它**把这条教训存进项目记忆文件**。久而久之，这个项目上的 AI 就越来越懂行。他强调：**不需要什么花哨的记忆系统，一个 markdown 文件就够**。

**② 技能（skill）**，用来给记忆文件"减肥"。

项目记忆久了会越来越臃肿。他的办法：**把"只在特定情况才用到"的内容挪进技能**。举例：一段"端到端测试怎么做"的说明，只有当 AI 要**改代码**时才用得上；如果你只是问它个问题，这段就是纯浪费 token。

技能的妙处在于**渐进披露（progressive disclosure）**：AI 启动时只把每个技能的**一行描述**读进系统提示，知道"有这么个技能、是干嘛的"；只有当它**真决定要用**某个技能时，才去读这个技能的完整内容。这样你能存海量"怎么做某事"的知识，却不会每次请求都为用不上的内容付 token。

> 怎么让 AI 学会自己建技能？装 Anthropic 写的 **skill creator** 技能；他用 Vercel 出的 **`npx skills`** 命令行工具来装和管理技能，几乎支持所有 agent。

### ⚠️ 一个重要警告：别乱装网上的高星技能

他专门花了一段讲这个，分两层：

1. **安全**：技能能指挥你的 AI 在你机器上跑**几乎任何命令**——可能在你不知情时把你的 API key、甚至银行凭据泄露给不可信的第三方。
2. **性能**：就算不谈安全，很多技能其实会**拖垮**你的 AI。他点名了那个有 **17.7 万星**的 "Andrej Karpathy skills" 仓库——星数巨多，看着肯定很强对吧？他用 ProgramBench（测 AI 端到端写程序的能力）实测：用了这个技能后，AI **多花 5% token，结果还更差**。而且这仓库**根本不是 Karpathy 本人写的**。

他的结论：**GitHub 星数只说明流行，不等于有用**。很多被疯传的技能，不过是某个人自己碰巧觉得有效、然后火了，从没经过严谨评测。**凡是号称"能神奇地让你的 AI 更强"、却拿不出严谨证据的技能，一律别装。**

## 第 3 级 · 带一个船员干活（这是全片的核心方法）

### 怎么"说话"：他几乎全程语音输入

不打字，直接说："用简洁的方式解释这个仓库，再给我讲讲最近几个 PR 在干嘛。" 有篇斯坦福论文正经比较过——**说话比打字快 3 倍**，这是实打实的效率提升。

- 转写用 **open-super-whisper**：免费开源、**在本地**跑 whisper 模型，质量很好（数据不出本机）。
- 唯一会退回打字的情况：要给 AI 一个 **URL 或文件路径**（你不会想把一串网址念出来）。
- 小技巧：把你常用的专有名词（项目名等）写进转写软件的 initial prompt，识别率立刻变准。

### 工具的"对 AI 友好度"差别巨大——这段最有工程含量

AI 干活要靠外部工具（比如调 GitHub 拉数据），**这些工具怎么设计，会极大影响 AI 的表现**。他拿 GitHub 举例并跑了 benchmark：

- 用大家常用的 **GitHub MCP server**，比直接用 GitHub **CLI** 要**多花 3 倍 token、2 倍多的延迟**——花了时间和钱却没明显好处。
- 他自己搞了套东西叫 **axi / ACI**：一套"把 AI 当一等公民"的工具设计标准，10 条原则。比如**用紧凑的输出格式，比用 JSON 省约 40% token**。他据此做了 GitHub axi、Chrome DevTools axi，benchmark 显示 AI 用更少轮次、更少 token 就把同样的活干完。

> 给你的可操作结论：**给 AI 配工具之前，先查一下这个工具对 AI 高不高效**，它能极大影响你从 AI 身上压榨出多少产能。

### 规划阶段：别让它甩给你一堵"文字墙"

很多人现在的默认流程是：让 AI 出个方案，它回你一大段文字（plan mode 或写进 markdown）。问题是：这段文字很难读，你**没法指着其中某一句说"这块不对"**，也没法选中一段给反馈。

他的工具 **Lavish** 解决这个：它让 AI 生成一个**用当前项目自身设计风格做的 HTML 原型页面**，在浏览器里打开。于是你能**直观看到每个方案长什么样**、能在具体某处**圈点批注**、底部的待定项**点一下就做决定**，反馈直接发回给 AI，连终端都不用切回去。他演示的真实例子：把他给孩子做的 AI 辅导 App 里两个功能重复的按钮合并成一个"成就系统"。他说看完 Lavish 再也回不去读终端里的文字墙了。

### 实现阶段：他基本不插手

因为需求在规划阶段已经聊清楚了，实现过程中他通常**完全不干预**，只等它说"做完了"。

### 验收阶段：别逐行审 diff——这是最重要的心态

AI 说"做完了"恰恰是最棘手的时刻。很多人这时打开编辑器开始审 diff。他的观点很尖锐：

> **AI 写代码太快了，如果每段代码都要你审，你就成了自己的瓶颈**——你一天能审的 diff 是有限的，你的产出速度会被这个上限死死卡住。而且，**审 diff 一点都不好玩**，没人是为了"整天审 diff"才当工程师的。

他的解法是换身份：**别把自己当"审代码的人"，要当"工程经理/总监"**。总监基本不审 PR，但能靠**定好流程和文化、让团队执行**来保证质量。对 AI 也该这样。

落地工具是他自建的开源流水线 **no mistakes**：AI 一说做完，他不审 diff、不手动测，直接把改动丢进这条流水线，它会自动：

1. 没分支就建分支，然后 commit；
2. 在一个**隔离的 worktree**（独立工作副本）里跑，全程不影响你当前仓库；
3. 先**分析你的 AI 会话、推断你这次改动的真实意图**；
4. rebase 到远端最新的 main，**提前解掉合并冲突**；
5. 在**一个全新的上下文窗口**里做**对抗式 review**——大部分问题在这一步被抓出并自我修正；但**有产品含义的模糊问题，会上报给你这个人来拍板**；
6. 按"你的原始意图"做**端到端测试**，并**录下证据**证明改动真的работает——证据可能是截图、录屏、日志；
7. 过一遍文档，把相关文档更新到最新；
8. 确认没有 lint 问题，推送分支、提 PR；
9. **一直盯着这个 PR 直到合并**——期间新进来的合并冲突、CI 失败这些烦人事，它都替你处理。

PR 出来后，他看的是它给的**风险评估**：决定自己要不要细看。**低风险改动他根本不看 diff**——因为反复验证下来，低风险里他能发现的问题，流水线基本都已经发现了；只有更高风险的改动才值得他花时间。这就是他**在保证质量的前提下，把每天的代码改动量做到很大**的办法。也可以把 `no mistakes` 当技能直接喊。

**结果是：他花时间的地方挪到了任务的开头（在 Lavish 里把需求想清楚）和结尾（把质量关），中间全交给 AI**——中间被腾出来的时间越多，他能并行去开的活就越多。

## 第 4 级 · 让 AI 干更久、并行干更多

### 怎么让 AI 一次干更久：good night have fun

中间想腾出更多时间，就得给 AI **更复杂、要花更久**的任务。极端版是：他晚上要睡 7–8 小时，怎么让 AI 这 8 小时一直有事干？

他的开源工具 **good night have fun**：给它一个目标 + 一个停止条件（token 上限 / 轮次上限 / 自定义条件），它就**循环干到达标为止**。演示的例子特别妙：

> "假装你是个 7 岁小孩，端到端地用这个 App（创建档案那步是给家长的，跳过）。在 App 里到处试，找出**第一个**会让你这个小孩困惑、或不知道下一步该怎么办的可用性问题。找到就停下来修掉，然后重复。"

他能看实时 token 消耗、跑了几轮、提交了几个 commit，然后安心去睡；早上回来挑哪些 commit 要留。这招适合**可验证的目标**：降低页面加载时间、提高端到端测试覆盖率，或者像 [[4-自我改进优化循环-karpathy-autoresearch]] 那样"不断试假设来优化某个指标"。

> 他提到 Codex 和 Claude Code 新出的 `/go` 命令也能做类似的事，但 good night have fun 更好用在于：能**精确设上限**——否则你睡前挂个目标，醒来可能发现一周的额度被烧光了。

### 怎么同时带好几个 AI：treehouse

同一个目录里跑两个 AI 会互相踩、起冲突。标准解法是 **git worktree**（给仓库建一份独立的工作副本，互不干扰）。但 worktree 有个麻烦：**你得在脑子里记账**——我建了哪些、这个 `hibit2` 上次在干嘛、里面还有没有 AI 在跑、用完还得手动删。这些全成了心智负担。

他的工具 **treehouse** 把这事自动化：在仓库里敲一下 `treehouse`，它直接把你丢进一个新的干净 worktree；再敲一次再开一个；`treehouse status` 看哪些在用哪些空闲；**关掉那个 tab，它就知道你用完了、自动释放**，下次优先复用空闲的而不是新建。

他演示同时开三个 tab、三个 AI 并行改三件事（语音输入按钮加提示、截图当附件、状态栏总显示 bot 活动），靠顶部状态栏判断哪个需要他、键盘快捷键在 tab 间切换。

## 第 5 级 · 请个"大副"，你才真正成为船长

并行带多个 AI 一阵子之后他发现：**在这些会话之间反复切换、还得不断提醒自己每个在干嘛，非常累**，这不像理想的终局体验。于是他需要一个"大副"——一个他能像船长一样**对话**的角色，替他执行所有指令、管理所有船员，让他能专注在"我们下一步该往哪走"这种大局上，而不是跟越来越多的 AI 打地鼠。

他的开源工具 **first mate** 就是这个大副：克隆下来、在里面跑个 agent，然后你就只管跟它说话。比如他说"给这三个仓库的命令行都加一个把版本更新到 npm 最新的命令"——first mate **自己意识到这是 3 个并行任务**，在幕后开 tmux tab、调 treehouse 建 worktree、在里面跑 AI 干活、再跑 no mistakes 验证、把 PR 备好等他 review。他还能继续甩活给它（"看看 lavish 仓库最近三个 open issue，讨论哪个能做"），它一边等后台三个 agent 一边去拉 issue。**所有的"调度杂耍"都由 first mate 做了，他只管不停给方向。**

有意思的收尾观察：有了大副之后，你会开始**想不出该让它干嘛了**。这是好事——说明瓶颈转移了，从"AI 不够快"变成了"**你这个船长跟不跟得上**"。这时候需要一次心态转变：**把精力更多花在搞清楚什么才重要上**——去跟用户聊、研究竞争格局、画一张能把船队带向正确方向的"藏宝图"。做到这一步，你就从水手变成了真正的船长。
