# 总结 Schema（Claude 读 clean/*.txt 时遵循）

> 读者是 10 年经验的软件工程师，默认已懂工程基础。
> 任务不是"复述视频"，而是替他决策"值不值得看原片"并萃取真正有信息量的部分。**狠过滤废话。**

**语言**：中英混合 —— 技术术语保留英文（MCP, agentic loop, context window），叙述用中文。

## 文件命名
- 笔记文件名用**可读 kebab-case slug**（取自标题精炼），不要用 video_id。
  例：`claude-code-6-best-skills.md`、`mcp-server-from-scratch.md`。
- video_id **写进正文**第二行（`video_id: xxx` + youtu.be 链接），判重靠它，不靠文件名。

## 排版原则（重要 —— 要清爽可扫，不要密集 bullet 墙）
- 顶部一个 **verdict 引用块**：emoji 圆点 + 总分 + WATCH/SKIM/SKIP + 一句话理由。
- 评分用**三行小表 + 星级**（★★★☆☆），不要写成 5 行 bullet。
- 多个工具/技术点优先用**表格**呈现（工具 | 是什么 | 解决什么），而非长 bullet。
- 每节短，少嵌套，少括号套括号。给眼睛留白（用 `---` 分隔）。

## 模板

```markdown
# <精炼中文标题>

> 一句话理由（值得看什么 / 为什么跳过）

| 维度 | 评级 |
|------|------|
| 新颖度 | ★★☆☆☆ <一句> |
| 可操作性 | ★★★★☆ <一句> |
| 深度 | ★★☆☆☆ <一句> |

---

## 📦 核心技术点 / 工具
<能用表格就用表格>

## ✅ 能直接搬进你工作流的
<可复用做法/配置/prompt/工具链；没有写"无">

## 💡 反共识 / 值得内化的判断框架
<作者哪里跟主流不同、推翻了什么假设 —— 高价值信号>

## ⚠️ 待自测
<视频里的数字/benchmark 来源，提醒是单方信源>

`#tag` `#tag`
```

## 归档规则
- 总分 ≥ 4 → `knowledge/`，index 标 🟢。
- 总分 3 → 留 `notes/`，index 标 🟡。
- 总分 ≤ 2 → `rejected/`，index 标 🔴 + 一行说明为什么不值得。
- 每次总结后更新 `index.md`（按评分排序的总表）。
