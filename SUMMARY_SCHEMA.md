# 总结 Schema（Claude 读 clean/*.txt 时遵循）

> 读者是 10 年经验的软件工程师，默认已懂工程基础。
> 任务不是"复述视频"，而是替他决策"值不值得看原片"并萃取真正有信息量的部分。**狠过滤废话。**

**语言**：中英混合 —— 技术术语保留英文（MCP, agentic loop, context window），叙述用中文。

## 篇幅：按价值分两档
- **常规（≤4 分）**：精简版，表格 + 短 bullet，几分钟读完。
- **深度版（5 分，或用户点名要）**：写成"能当文章读的走读版"——按视频自身结构分节，每节用自然语言把"是什么 + 为什么 + 例子"讲透，多用解释性段落少用电报体。详尽但仍口语、仍按可读性铁律。文件名 tag 加 `#深度版`。

## 文件命名与存放
- **所有笔记都放在 `notes/` 一个目录**（不再按分数拆 knowledge/rejected 文件夹）。
- 文件名格式：**`<评分>-<简短中文>-<english-slug>.md`**，评分开头便于排序，中英双语便于辨认。
  例：`4-第二大脑AI操作系统-claude-second-brain-ai-os.md`、`3-最值得装的6个skill-claude-code-6-best-skills.md`。
- video_id **写进正文**第二行（`video_id: xxx` + youtu.be 链接），判重靠它，不靠文件名。

## 排版原则（重要 —— 要清爽可扫，不要密集 bullet 墙）
- 顶部一个 **verdict 引用块**：emoji 圆点 + 总分 + WATCH/SKIM/SKIP + 一句话理由。
- 评分用**三行小表 + 星级**（★★★☆☆），不要写成 5 行 bullet。维度名也用大白话（"能不能上手用""新鲜感"），别写"可操作性/novelty"。
- 多个工具/技术点优先用**表格**呈现，列名口语化（如"一句话说它是什么 | 帮你解决的麻烦"）。
- 每节短，少嵌套，少括号套括号。给眼睛留白（用 `---` 分隔）。

## 可读性铁律（**全篇都要，不只开头**）
开头大白话、正文却是术语墙 = 不合格。整篇每一条都按这个来：
- **每条先用人话给结论，再补技术细节**。不要写电报体（"路由式 CLAUDE.md、scoped key 都能直接用"这种堆词不行）。
- **术语第一次出现就地解释一句**：写成「人话（术语名）」或「术语 —— 一句解释」。
  例：`AI 会开始忘事、糊弄（业内叫 context rot）`、`限定权限的钥匙（scoped key）`、`记忆额度（token）`。
- **工具/产品名保留英文**（Superpowers、GSD、Sonnet 等，要对得上原视频），但紧跟一句它干嘛。
- 给资深读者写，但要让他**扫一眼就懂**，而不是逐词解码。技术深度不削，只是把话说顺。

## 归档规则
- 所有笔记进 `notes/`，文件名以评分开头（见上）。高/低价值用 **index.md 的评分 emoji** 区分，不靠分文件夹。
